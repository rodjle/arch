Sistema de Avaliação de Trabalhos

📋 O QUE O SISTEMA FAZ?

Este é um sistema educacional que permite avaliar trabalhos apresentados por alunos. Funciona assim:

👤 Alunos
São cadastrados no sistema com ID e nome
📝 Apresentação de Trabalhos
Um aluno apresenta um trabalho (ex: "Arquitetura em Camadas")
⭐ Avaliação por Colegas
Outros alunos dão notas de 0 a 10 para o trabalho apresentado
→ Validação: a nota deve estar entre 0 e 10 (regra de negócio!)
📊 Cálculo da Média
O sistema calcula a média das notas recebidas por um trabalho
→ Regra: soma das notas ÷ quantidade de avaliações
🎯 EXEMPLO PRÁTICO (rodando no código):
  1. Alice apresenta o trabalho "Arquitetura em Camadas"
  2. Bob avalia e dá nota 8.5
  3. Carol avalia e dá nota 9.0
  4. Alice (pode avaliar o próprio trabalho) dá nota 7.0
  5. Sistema calcula a média: (8.5 + 9.0 + 7.0) ÷ 3 = 8.17 ✅

☠️ PROBLEMA DA ARQUITETURA EM CAMADAS CLÁSSICA

OLHE NO DIAGRAMA ACIMA: As setas VERMELHAS TRACEJADAS com símbolo ☠️ mostram o problema!

O que está acontecendo?
O GerenciadorAvaliacoes (camada Application - verde) está chamando diretamente as classes concretas AlunoRepositorio e TrabalhoRepositorio (camada Infrastructure - vermelha).

Por que isso é um problema?

❌ Acoplamento Forte Se você quiser trocar o AlunoRepositorio (que usa HashMap) por um AlunoRepositorioJDBC (que usa banco de dados), você precisa modificar o código de quem cria o GerenciadorAvaliacoes.
❌ Dificulta Testes Para testar o GerenciadorAvaliacoes isoladamente, você é obrigado a instanciar os repositórios reais. Não consegue criar um "repositório falso" (mock) facilmente.
❌ Viola DIP O Dependency Inversion Principle diz: "módulos de alto nível não devem depender de módulos de baixo nível; ambos devem depender de abstrações". Aqui, Application (alto nível) depende diretamente de Infrastructure (baixo nível).
❌ Conhecimento Técnico A camada Application "sabe" que existem classes concretas AlunoRepositorio e TrabalhoRepositorio. Ela conhece detalhes da Infrastructure, violando a separação limpa de responsabilidades.

✅ SOLUÇÃO: Arquitetura Hexagonal (Ports & Adapters)

Como resolver esse problema?

  1. Criar interfaces (Ports) no Domain:
    • interface IAlunoRepositorio { Aluno buscarPorId(String id); void salvar(Aluno a); }
    • interface ITrabalhoRepositorio { Trabalho buscarPorId(String id); void salvar(Trabalho t); }
  2. O GerenciadorAvaliacoes (Application) passa a depender das interfaces, não das implementações:
    • public GerenciadorAvaliacoes(IAlunoRepositorio repo, ...)
  3. As classes AlunoRepositorio e TrabalhoRepositorio (Infrastructure) implementam as interfaces
  4. Resultado: A seta de dependência é invertida! Infrastructure passa a depender de Domain (interfaces), e Application também depende apenas de Domain.

🎯 Na Hexagonal Architecture, esse problema NÃO existe — você verá na próxima arquitetura!
