🏛️ O QUE É CLEAN ARCHITECTURE?

Clean Architecture é uma arquitetura de software criada por Robert C. Martin (Uncle Bob) que organiza o código em 4 camadas concêntricas, com a regra fundamental de que as dependências sempre apontam para dentro.

📚 Origem

Proposta no livro "Clean Architecture: A Craftsman's Guide to Software Structure and Design" (2017), a Clean Architecture é uma evolução e unificação de várias arquiteturas anteriores: Hexagonal Architecture, Onion Architecture, DCI, BCE, entre outras.

🎯 Objetivos Principais

  • Independência de Frameworks: A arquitetura não depende de bibliotecas externas
  • Testabilidade: Regras de negócio podem ser testadas sem UI, banco, servidor, etc
  • Independência de UI: A interface pode mudar sem afetar o sistema
  • Independência de Banco de Dados: Regras de negócio não dependem do BD
  • Independência de Agentes Externos: Regras de negócio não sabem nada sobre o mundo externo

✅ A REGRA DE DEPENDÊNCIA

⚠️ REGRA MAIS IMPORTANTE DA CLEAN ARCHITECTURE

"Source code dependencies must point only inward, toward higher-level policies."

Tradução: As dependências do código-fonte devem apontar apenas para dentro, em direção a políticas de nível mais alto.

🔍 O que isso significa?

  • Entities não dependem de nada (centro do círculo)
  • Use Cases dependem apenas de Entities
  • Interface Adapters dependem de Use Cases e Entities
  • Frameworks & Drivers dependem de todas as camadas internas

🚫 O que NÃO pode acontecer?

  • Entities importarem Use Cases (❌)
  • Use Cases importarem Controllers (❌)
  • Camadas internas conhecerem detalhes de camadas externas (❌)
🔴 → 🟠 → 🟢 → 🟡

Dependências SEMPRE apontam para DENTRO!

🔄 Como inverter dependências?

Use o Dependency Inversion Principle (DIP):

  • Camadas internas definem interfaces
  • Camadas externas implementam essas interfaces
  • Isso faz as camadas externas dependerem das internas!

✅ Exemplo em nosso sistema:

IAlunoGateway é DEFINIDA nos Use Cases (camada 2)

AlunoGatewayMemoria IMPLEMENTA essa interface (camada 3)

Resultado: Adapters dependem de Use Cases, não o contrário! 🎉

📐 AS 4 CAMADAS DA CLEAN ARCHITECTURE

🟡 Camada 1: ENTITIES (Enterprise Business Rules)

Responsabilidade:

Encapsula as regras de negócio mais críticas e fundamentais da empresa. São as regras que existiriam mesmo sem um sistema de software.

Características:

  • Não depende de NADA (centro do círculo)
  • Contém objetos de negócio fundamentais
  • Mudanças nesta camada são raras
  • Pode ser compartilhada entre múltiplas aplicações

Exemplos no nosso sistema:

  • Aluno - Regra: nome não pode ser vazio
  • Nota - Regra: nota deve estar entre 0 e 10
  • Trabalho - Regra: cálculo de média

🟢 Camada 2: USE CASES (Application Business Rules)

Responsabilidade:

Orquestra o fluxo de dados entre Entities e o mundo externo. Contém as regras específicas desta aplicação.

Características:

  • Depende apenas de Entities
  • Cada Use Case é uma classe separada (princípio SRP)
  • Define interfaces de Gateways (inversão de dependência)
  • Usa DTOs (Input/Output) nas fronteiras

Exemplos no nosso sistema:

  • CadastrarTrabalhoUseCase - Orquestra cadastro de trabalho
  • AvaliarTrabalhoUseCase - Orquestra avaliação
  • ConsultarMediaUseCase - Orquestra consulta

🟠 Camada 3: INTERFACE ADAPTERS

Responsabilidade:

Converte dados entre o formato dos Use Cases e o formato externo. Adaptadores que traduzem entre camadas.

Três tipos de Adapters:

  • Controllers: Recebem requisições, criam Inputs, chamam Use Cases
  • Presenters: Recebem Outputs, formatam para visualização
  • Gateways: Implementam interfaces de persistência (definidas nos Use Cases)

Exemplos no nosso sistema:

  • AvaliacaoController - Controla fluxo da aplicação
  • ConsolePresenter - Formata saída para console
  • AlunoGatewayMemoria - Implementa persistência em memória
  • TrabalhoGatewayMemoria - Implementa persistência em memória

🔴 Camada 4: FRAMEWORKS & DRIVERS

Responsabilidade:

Camada mais externa. Frameworks, ferramentas, drivers, e o ponto de entrada (Main).

Características:

  • Camada mais volátil (muda mais frequentemente)
  • Contém detalhes de implementação
  • Main.java faz a composição (Dependency Injection)
  • Conhece TODAS as camadas (único permitido)

Exemplos no nosso sistema:

  • Main.java - Ponto de entrada e DI manual
  • Console I/O
  • HashMap (simulando banco de dados)

🔄 DIFERENÇAS ENTRE HEXAGONAL E CLEAN

Aspecto Hexagonal Architecture Clean Architecture
Metáfora Hexágono com portas (Ports) Círculos concêntricos
Camadas 3 camadas: Domain, Application, Adapters 4 camadas: Entities, Use Cases, Interface Adapters, Frameworks
Foco Independência de tecnologia (I/O) Separação de responsabilidades por camada
Use Cases Application Service único (ex: GerenciadorAvaliacoes) Classe separada por Use Case (CadastrarTrabalhoUseCase)
DTOs Opcional Fortemente recomendado (Input/Output)
Nomenclatura Ports (input/output), Adapters (driving/driven) Use Cases, Gateways, Presenters, Controllers
Regra Inversão de Dependência (DIP) Regra de Dependência (sempre para dentro)
Controller Pode ter main() no Controller Main separado em Frameworks & Drivers
Saída Controller geralmente formata saída Presenter separado para formatação
Persistência Repository (nomenclatura comum) Gateway (nomenclatura preferida)

🤝 Semelhanças

  • Ambas aplicam Inversão de Dependência (DIP)
  • Ambas usam interfaces nas fronteiras
  • Ambas buscam independência de frameworks
  • Ambas facilitam testes automatizados
  • Ambas protegem o domínio/negócio de detalhes técnicos

🤔 Quando usar cada uma?

Use Hexagonal Architecture quando:

  • Sistema mais simples
  • Foco principal em desacoplamento tecnológico (trocar BD, UI)
  • Equipe pequena, precisa de simplicidade
  • Aplicações com poucos casos de uso

Use Clean Architecture quando:

  • Sistema complexo com muitos casos de uso
  • Foco em separação clara de responsabilidades
  • Equipe grande, precisa de organização rígida
  • Necessidade de granularidade (um Use Case por operação)
  • Projeto de longa duração que precisa de manutenibilidade

🔄 FLUXO COMPLETO: Main → Controller → Use Case → Gateway → Entity

Vamos acompanhar o fluxo completo de uma operação, desde o Main.java até as Entities:

📍 Passo 1: Main.java (Frameworks & Drivers)

// Main instancia TUDO e faz a "composição"
IAlunoGateway alunoGateway = new AlunoGatewayMemoria();
ITrabalhoGateway trabalhoGateway = new TrabalhoGatewayMemoria();

CadastrarTrabalhoUseCase useCase = new CadastrarTrabalhoUseCase(
    trabalhoGateway,  // Injeção de dependência
    alunoGateway      // através de interfaces
);

ConsolePresenter presenter = new ConsolePresenter();

AvaliacaoController controller = new AvaliacaoController(
    useCase, ..., presenter, alunoGateway
);

controller.executar();  // Inicia o fluxo

📍 Passo 2: Controller (Interface Adapter)

// Controller prepara os dados e chama Use Case
CadastrarTrabalhoInput input = new CadastrarTrabalhoInput(
    "T001",
    "Clean Architecture na Prática",
    "A001"
);

CadastrarTrabalhoOutput output = useCase.executar(input);

presenter.apresentarCadastroTrabalho(output);

📍 Passo 3: Use Case (Application Business Rules)

// Use Case orquestra o fluxo
public CadastrarTrabalhoOutput executar(CadastrarTrabalhoInput input) {
    // 1. Busca aluno via Gateway (interface)
    Aluno autor = alunoGateway.buscarPorId(input.getAlunoId());
    
    // 2. Cria Entity (regras de negócio são validadas)
    Trabalho trabalho = new Trabalho(
        input.getTrabalhoId(),
        input.getTitulo(),
        autor
    );
    
    // 3. Salva via Gateway
    trabalhoGateway.salvar(trabalho);
    
    // 4. Retorna Output
    return CadastrarTrabalhoOutput.sucesso(...);
}

📍 Passo 4: Gateway (Interface Adapter)

// Gateway implementa a interface definida nos Use Cases
public class TrabalhoGatewayMemoria implements ITrabalhoGateway {
    private Map<String, Trabalho> trabalhos = new HashMap<>();
    
    @Override
    public void salvar(Trabalho trabalho) {
        trabalhos.put(trabalho.getId(), trabalho);
    }
}

📍 Passo 5: Entity (Enterprise Business Rules)

// Entity contém as regras de negócio
public class Trabalho {
    private final String id;
    private final String titulo;
    private final Aluno autor;
    private final List<Nota> notas;
    
    public Trabalho(String id, String titulo, Aluno autor) {
        // Validações de negócio aqui
        this.id = id;
        this.titulo = titulo;
        this.autor = autor;
        this.notas = new ArrayList<>();
    }
    
    public double calcularMedia() {
        // REGRA DE NEGÓCIO: Cálculo de média
        if (notas.isEmpty()) return 0.0;
        return notas.stream()
                    .mapToDouble(Nota::getValor)
                    .average()
                    .orElse(0.0);
    }
}

📍 Passo 6: Presenter (Interface Adapter)

// Presenter formata o Output para exibição
public void apresentarCadastroTrabalho(CadastrarTrabalhoOutput output) {
    System.out.println("===================================");
    System.out.println("RESULTADO: CADASTRAR TRABALHO");
    System.out.println("===================================");
    System.out.println(output.getMensagem());
    
    if (output.isSucesso()) {
        System.out.println("ID: " + output.getTrabalhoId());
        System.out.println("Título: " + output.getTitulo());
        System.out.println("Autor: " + output.getNomeAutor());
    }
}

🎯 Inversão de Dependência em Ação

Observe como as dependências são invertidas:

  • ITrabalhoGateway é DEFINIDA nos Use Cases (camada 2)
  • TrabalhoGatewayMemoria IMPLEMENTA essa interface (camada 3)
  • Use Case depende da INTERFACE (abstração)
  • Gateway implementa a INTERFACE (concretização)
  • Resultado: Camada 3 depende da Camada 2, não o contrário!

🧪 EXEMPLO DE TESTE UNITÁRIO DE USE CASE

Uma das grandes vantagens da Clean Architecture é a testabilidade. Use Cases podem ser testados isoladamente com mocks dos Gateways:

import org.junit.jupiter.api.Test;
import org.mockito.Mockito;
import static org.junit.jupiter.api.Assertions.*;
import static org.mockito.Mockito.*;

public class CadastrarTrabalhoUseCaseTest {
    
    @Test
    public void deveCadastrarTrabalhoComSucesso() {
        // ARRANGE: Preparar
        
        // Mock dos Gateways (não precisamos de implementações reais!)
        IAlunoGateway alunoGateway = mock(IAlunoGateway.class);
        ITrabalhoGateway trabalhoGateway = mock(ITrabalhoGateway.class);
        
        // Configurar comportamento do mock
        Aluno autor = new Aluno("A001", "João Silva");
        when(alunoGateway.buscarPorId("A001")).thenReturn(autor);
        when(trabalhoGateway.existe("T001")).thenReturn(false);
        
        // Instanciar Use Case (injetando mocks)
        CadastrarTrabalhoUseCase useCase = new CadastrarTrabalhoUseCase(
            trabalhoGateway,
            alunoGateway
        );
        
        // Criar Input
        CadastrarTrabalhoInput input = new CadastrarTrabalhoInput(
            "T001",
            "Clean Architecture na Prática",
            "A001"
        );
        
        // ACT: Executar
        CadastrarTrabalhoOutput output = useCase.executar(input);
        
        // ASSERT: Verificar
        assertTrue(output.isSucesso());
        assertEquals("T001", output.getTrabalhoId());
        assertEquals("Clean Architecture na Prática", output.getTitulo());
        assertEquals("João Silva", output.getNomeAutor());
        
        // Verificar que os mocks foram chamados corretamente
        verify(alunoGateway, times(1)).buscarPorId("A001");
        verify(trabalhoGateway, times(1)).existe("T001");
        verify(trabalhoGateway, times(1)).salvar(any(Trabalho.class));
    }
    
    @Test
    public void deveRetornarErroQuandoAlunoNaoExistir() {
        // ARRANGE
        IAlunoGateway alunoGateway = mock(IAlunoGateway.class);
        ITrabalhoGateway trabalhoGateway = mock(ITrabalhoGateway.class);
        
        // Configurar mock para retornar null (aluno não existe)
        when(alunoGateway.buscarPorId("A999")).thenReturn(null);
        
        CadastrarTrabalhoUseCase useCase = new CadastrarTrabalhoUseCase(
            trabalhoGateway,
            alunoGateway
        );
        
        CadastrarTrabalhoInput input = new CadastrarTrabalhoInput(
            "T001",
            "Teste",
            "A999"  // Aluno inexistente
        );
        
        // ACT
        CadastrarTrabalhoOutput output = useCase.executar(input);
        
        // ASSERT
        assertFalse(output.isSucesso());
        assertTrue(output.getMensagem().contains("não encontrado"));
        
        // Verificar que salvar NÃO foi chamado
        verify(trabalhoGateway, never()).salvar(any());
    }
}

✅ Vantagens dos Testes em Clean Architecture:

  • Use Cases testáveis isoladamente: Não precisa de banco de dados, servidor, ou UI
  • Mocks simples: Use Cases dependem de interfaces, fácil de mockar
  • Testes rápidos: Não há I/O, apenas lógica pura
  • Cobertura alta: Pode testar todos os cenários (sucesso, erros, validações)
  • Confiança: Se os testes passam, as regras de negócio estão corretas

🎯 O que mais podemos testar?

  • Entities: Testar regras de negócio (validações, cálculos)
  • Input DTOs: Testar validações de formato
  • Output DTOs: Testar factory methods
  • Gateways: Testar com banco de dados em memória (H2, SQLite)
  • Controllers: Testar com mocks de Use Cases

🎓 CONCLUSÃO

✅ O que aprendemos sobre Clean Architecture?

  • Organização em 4 camadas concêntricas
  • A Regra de Dependência (sempre para dentro)
  • Use Cases granulares (um por operação)
  • DTOs nas fronteiras (Input/Output)
  • Inversão de Dependência com interfaces
  • Main separado na camada externa
  • Presenter separado do Controller

🎯 Princípios SOLID aplicados:

Princípio Como é aplicado
S - Single Responsibility Cada Use Case faz uma coisa só. Presenter só formata. Entity só tem regras de negócio.
O - Open/Closed Pode adicionar novos Use Cases sem modificar existentes. Pode adicionar novos Gateways.
L - Liskov Substitution Qualquer implementação de IAlunoGateway pode substituir outra.
I - Interface Segregation Interfaces pequenas e específicas (IAlunoGateway, ITrabalhoGateway).
D - Dependency Inversion Use Cases dependem de interfaces, não de implementações. Adapters implementam interfaces.

🚀 Quando usar Clean Architecture?

✅ Use quando:

  • Projeto de longa duração (anos)
  • Equipe grande que precisa de estrutura clara
  • Muitos casos de uso diferentes
  • Alta necessidade de testes automatizados
  • Precisa trocar tecnologias frequentemente

❌ Não use quando:

  • Projeto muito simples (CRUD básico)
  • Prototipo ou MVP rápido
  • Equipe muito pequena (1-2 pessoas)
  • Tempo de desenvolvimento curtíssimo
