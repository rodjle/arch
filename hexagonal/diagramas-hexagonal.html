🏛️ Arquitetura Hexagonal

Sistema de Avaliação de Trabalhos Acadêmicos

Ports & Adapters Pattern

📚 O que é Arquitetura Hexagonal?

A Arquitetura Hexagonal (também conhecida como Ports & Adapters) é um padrão arquitetural criado por Alistair Cockburn que busca isolar a lógica de negócio do sistema de seus mecanismos de entrega e infraestrutura.

🔷 Conceito Principal

O núcleo da aplicação (Domain + Application) é representado por um hexágono que possui portas (interfaces) através das quais adaptadores (implementações) podem se conectar.

  • 🧱 Portas (Ports): Interfaces que definem pontos de entrada e saída do hexágono
  • 🔌 Adaptadores (Adapters): Implementações concretas que se conectam às portas
  • 💎 Núcleo (Core): Domain + Application, 100% independente de tecnologias externas

🎯 Objetivo

Criar um sistema onde a lógica de negócio seja completamente independente de:

  • Frameworks (Spring, Jakarta EE, etc.)
  • Bancos de dados (SQL, NoSQL, arquivos, etc.)
  • APIs externas
  • Interfaces de usuário (Web, Console, Mobile, etc.)

❌ Problema na Arquitetura em Camadas

🔴 Situação Anterior

Na arquitetura em camadas tradicional, a camada Application dependia DIRETAMENTE das classes concretas da camada Infrastructure.

Estrutura Antiga (Camadas):

src/
├── presentation/
│   └── AvaliacaoController.java  (ponto de entrada)
├── application/
│   └── GerenciadorAvaliacoes.java  ❌ new TrabalhoRepositorio()
├── domain/
│   ├── Aluno.java
│   ├── Nota.java
│   └── Trabalho.java
└── infrastructure/
    ├── AlunoRepositorio.java  (HashMap - classe concreta)
    └── TrabalhoRepositorio.java  (HashMap - classe concreta)

🐛 Problemas Identificados:

Problema Consequência
❌ Acoplamento Forte Application conhece detalhes de infraestrutura (HashMap)
❌ Difícil Testar Não dá para passar mocks ou stubs facilmente
❌ Inflexível Trocar HashMap por SQL requer alterar Application
❌ Viola DIP Módulo de alto nível depende de módulo de baixo nível

Código Problemático:

// ❌ ANTES (Arquitetura em Camadas)
public class GerenciadorAvaliacoes {
    private TrabalhoRepositorio trabalhoRepo;  // ❌ Classe concreta!
    private AlunoRepositorio alunoRepo;        // ❌ Classe concreta!
    
    public GerenciadorAvaliacoes() {
        // ❌ Instancia diretamente (acoplamento)
        trabalhoRepo = new TrabalhoRepositorio();
        alunoRepo = new AlunoRepositorio();
    }
}

// Consequências:
// - Não dá para trocar por AlunoRepositorioJDBC sem alterar código
// - Não dá para passar mock nos testes
// - Application conhece que usa HashMap (detalhe de infraestrutura)

✅ Solução com Arquitetura Hexagonal

🟢 Nova Estrutura

Com Hexagonal, invertemos as dependências: Application depende de interfaces (Ports), e os Adapters implementam essas interfaces.

Nova Estrutura (Hexagonal):

src/
├── adapters/
│   ├── input/
│   │   └── AvaliacaoController.java  (Driving Adapter - Console)
│   └── output/
│       ├── AlunoRepositorioMemoria.java  (Driven Adapter - implementa IAlunoRepositorio)
│       └── TrabalhoRepositorioMemoria.java  (Driven Adapter - implementa ITrabalhoRepositorio)
├── application/
│   └── GerenciadorAvaliacoes.java  ✅ implements IGerenciadorAvaliacoes
└── domain/
    ├── model/
    │   ├── Aluno.java  (entidade)
    │   ├── Nota.java  (entidade)
    │   └── Trabalho.java  (entidade)
    └── ports/
        ├── input/
        │   └── IGerenciadorAvaliacoes.java  (Driving Port - interface de entrada)
        └── output/
            ├── IAlunoRepositorio.java  (Driven Port - interface de saída)
            └── ITrabalhoRepositorio.java  (Driven Port - interface de saída)

🔌 Exemplo Didático: O que o Adapter faz?

Pense no Adapter como um tradutor. O núcleo do sistema fala pela porta (interface), e o Adapter traduz isso para uma implementação concreta.

Elemento No exemplo didático No nosso sistema
Porta (interface) TomadaBrasileira IAlunoRepositorio
Implementação concreta PlugAmericano AlunoRepositorioMemoria
Adapter AdaptadorTomada AlunoRepositorioMemoria adaptando a porta
Cliente Carregador GerenciadorAvaliacoes

❌ Sem Adapter

// O carregador espera este contrato
interface TomadaBrasileira {
    void fornecerEnergia();
}

// Mas eu só tenho isso
class PlugAmericano {
    public void connectUSPlug() {
        System.out.println("Ligado no padrão americano");
    }
}

class Carregador {
    private TomadaBrasileira tomada;

    public Carregador(TomadaBrasileira tomada) {
        this.tomada = tomada;
    }
}

// ❌ Não compila:
// new Carregador(new PlugAmericano());

✅ Com Adapter

interface TomadaBrasileira {
    void fornecerEnergia();
}

class PlugAmericano {
    public void connectUSPlug() {
        System.out.println("Ligado no padrão americano");
    }
}

// Adapter: traduz uma interface para outra
class AdaptadorTomada implements TomadaBrasileira {
    private PlugAmericano plug;

    public AdaptadorTomada(PlugAmericano plug) {
        this.plug = plug;
    }

    public void fornecerEnergia() {
        plug.connectUSPlug();
    }
}

class Carregador {
    private TomadaBrasileira tomada;

    public Carregador(TomadaBrasileira tomada) {
        this.tomada = tomada;
    }

    public void carregar() {
        tomada.fornecerEnergia();
    }
}

// ✅ Agora funciona:
Carregador carregador = new Carregador(
    new AdaptadorTomada(new PlugAmericano())
);
carregador.carregar();

🎯 Ligando com a arquitetura hexagonal

No sistema, o GerenciadorAvaliacoes quer conversar com o contrato IAlunoRepositorio. Ele não quer saber se os dados vêm de memória, banco, arquivo ou API.

O papel do Adapter é exatamente esse: encaixar uma implementação concreta na porta que o núcleo espera. Assim, AlunoRepositorioMemoria se conecta ao núcleo implementando IAlunoRepositorio. 

// Porta esperada pelo núcleo
interface IAlunoRepositorio {
    Aluno buscarPorId(String id);
}

// Adapter concreto na borda do sistema
class AlunoRepositorioMemoria implements IAlunoRepositorio {
    public Aluno buscarPorId(String id) {
        // busca em HashMap, banco, arquivo etc.
    }
}

// Núcleo usa só a porta
class GerenciadorAvaliacoes {
    private IAlunoRepositorio alunoRepo;
}

✨ Benefícios Obtidos:

Benefício Como Funciona
✅ Inversão de Dependência Application depende de INTERFACES, não de classes concretas
✅ Fácil de Testar Injetar mocks das interfaces no construtor do Application
✅ Flexibilidade Trocar HashMap por SQL sem alterar uma linha no Application
✅ Respeita SOLID Dependency Inversion Principle (DIP) aplicado corretamente

Código Melhorado:

// ✅ DEPOIS (Arquitetura Hexagonal)
public class GerenciadorAvaliacoes implements IGerenciadorAvaliacoes {
    private ITrabalhoRepositorio trabalhoRepo;  // ✅ Interface!
    private IAlunoRepositorio alunoRepo;        // ✅ Interface!
    
    // ✅ Recebe dependências via construtor (Dependency Injection)
    public GerenciadorAvaliacoes(ITrabalhoRepositorio trabalhoRepo, 
                                  IAlunoRepositorio alunoRepo) {
        this.trabalhoRepo = trabalhoRepo;
        this.alunoRepo = alunoRepo;
    }
}

// No Controller (Driving Adapter):
// ✅ Controller decide qual implementação usar
IAlunoRepositorio alunoRepo = new AlunoRepositorioMemoria();

// ✅ Para produção, trocar por:
// IAlunoRepositorio alunoRepo = new AlunoRepositorioJDBC(conexao);

// ✅ Para testes, injetar mock:
// IAlunoRepositorio alunoRepo = mock(IAlunoRepositorio.class);

IGerenciadorAvaliacoes gerenciador = new GerenciadorAvaliacoes(trabalhoRepo, alunoRepo);
// ✅ Application não sabe qual implementação está usando!

💡 Pontos-Chave no Fluxo

  1. Controller chama IGerenciadorAvaliacoes (interface), não a classe concreta
  2. Application usa ITrabalhoRepositorio (interface), não a implementação em memória
  3. Polimorfismo em ação: JVM resolve qual implementação usar em runtime
  4. Fácil trocar: Mudar AlunoRepositorioMemoria para AlunoRepositorioJDBC → sem alterar Application!

🔄 Benefícios da Inversão de Dependência

🎯 Dependency Inversion Principle (DIP)

"Módulos de alto nível não devem depender de módulos de baixo nível. Ambos devem depender de abstrações."

1️⃣ Facilidade para Trocar Implementações

// No AvaliacaoController (Driving Adapter):

// === DESENVOLVIMENTO (rápido, volátil) ===
IAlunoRepositorio alunoRepo = new AlunoRepositorioMemoria();

// === PRODUÇÃO (persistente, escalável) ===
Connection conexao = DriverManager.getConnection("jdbc:postgresql://...");
IAlunoRepositorio alunoRepo = new AlunoRepositorioJDBC(conexao);

// === TESTES (mock, isolado) ===
IAlunoRepositorio alunoRepo = mock(IAlunoRepositorio.class);
when(alunoRepo.buscarPorId("a1")).thenReturn(new Aluno("a1", "Mock"));

// ===  APPLICATION NÃO MUDA! ===
IGerenciadorAvaliacoes gerenciador = new GerenciadorAvaliacoes(trabalhoRepo, alunoRepo);
gerenciador.cadastrarTrabalho(...);  // Funciona com qualquer implementação!

2️⃣ Testabilidade Simplificada

Com interfaces, criar testes unitários é trivial:

@Test
public void deveAvaliarTrabalhoCorretamente() {
    // 1. Criar mocks das interfaces (fácil!)
    ITrabalhoRepositorio mockTrabRepo = mock(ITrabalhoRepositorio.class);
    IAlunoRepositorio mockAlunoRepo = mock(IAlunoRepositorio.class);
    
    // 2. Configurar comportamento dos mocks
    Aluno autor = new Aluno("a1", "Alice");
    Aluno avaliador = new Aluno("a2", "Bob");
    Trabalho trabalho = new Trabalho("t1", "Título", autor);
    
    when(mockTrabRepo.buscarPorId("t1")).thenReturn(trabalho);
    when(mockAlunoRepo.buscarPorId("a2")).thenReturn(avaliador);
    
    // 3. Injetar mocks no Application (inversão de dependência!)
    IGerenciadorAvaliacoes gerenciador = new GerenciadorAvaliacoes(mockTrabRepo, mockAlunoRepo);
    
    // 4. Testar caso de uso SEM banco de dados real!
    gerenciador.avaliarTrabalho("n1", "t1", "a2", 8.5);
    
    // 5. Verificar que o método foi chamado
    verify(mockTrabRepo).salvar(trabalho);
    
    // ✅ Teste rápido, isolado, sem dependências externas!
}

3️⃣ Múltiplas Implementações Coexistem

Podemos ter várias implementações da mesma interface:

  • AlunoRepositorioMemoria: HashMap (desenvolvimento/testes)
  • AlunoRepositorioJDBC: PostgreSQL (produção)
  • AlunoRepositorioJPA: Hibernate/JPA (enterprise)
  • AlunoRepositorioMongo: MongoDB (NoSQL)
  • AlunoRepositorioCache: Redis (cache distribuído)
  • AlunoRepositorioREST: API externa (integração)

O melhor: Application não muda! Só mudamos qual Adapter injetamos.

4️⃣ Decorator Pattern (Empilhar Funcionalidades)

Podemos criar Adapters que decoram outros Adapters:

// Adapter com cache (decora outro adapter)
public class AlunoRepositorioComCache implements IAlunoRepositorio {
    private IAlunoRepositorio delegado;  // Outro adapter!
    private Cache cache;
    
    public AlunoRepositorioComCache(IAlunoRepositorio delegado, Cache cache) {
        this.delegado = delegado;
        this.cache = cache;
    }
    
    @Override
    public Aluno buscarPorId(String id) {
        // 1. Tentar buscar no cache
        Aluno cached = cache.get("aluno:" + id);
        if (cached != null) {
            System.out.println("✅ Cache HIT!");
            return cached;
        }
        
        // 2. Se não tiver no cache, buscar no repositório real
        System.out.println("❌ Cache MISS, buscando no DB...");
        Aluno aluno = delegado.buscarPorId(id);
        
        // 3. Armazenar no cache para próximas buscas
        if (aluno != null) {
            cache.put("aluno:" + id, aluno, 300);  // TTL 5 min
        }
        
        return aluno;
    }
    
    // ... outros métodos delegam para 'delegado' ...
}

// No Controller, empilhar Adapters:
IAlunoRepositorio repoBasico = new AlunoRepositorioJDBC(conexao);
IAlunoRepositorio repoComCache = new AlunoRepositorioComCache(repoBasico, redisCache);
IAlunoRepositorio repoComLog = new AlunoRepositorioComLogging(repoComCache);

IGerenciadorAvaliacoes gerenciador = new GerenciadorAvaliacoes(trabalhoRepo, repoComLog);
// ✅ Application não sabe que tem cache + logging! Transparente!

🧪 Exemplo de Teste Unitário

Um dos maiores benefícios da Arquitetura Hexagonal é a facilidade de testes. Veja como é simples testar o Application sem depender de banco de dados:

Teste com Mocks (Mockito):

import org.junit.jupiter.api.Test;
import static org.mockito.Mockito.*;
import static org.junit.jupiter.api.Assertions.*;

public class GerenciadorAvaliacoesTest {
    
    @Test
    public void deveCalcularMediaDeDuasNotasCorretamente() {
        // === ARRANGE (Preparar) ===
        
        // 1. Criar mocks das interfaces (simples!)
        ITrabalhoRepositorio mockTrabRepo = mock(ITrabalhoRepositorio.class);
        IAlunoRepositorio mockAlunoRepo = mock(IAlunoRepositorio.class);
        
        // 2. Criar entidades de teste
        Aluno autor = new Aluno("a1", "Alice");
        Aluno avaliador1 = new Aluno("a2", "Bob");
        Aluno avaliador2 = new Aluno("a3", "Carol");
        Trabalho trabalho = new Trabalho("t1", "Arquitetura Hexagonal", autor);
        
        // 3. Configurar comportamento dos mocks
        when(mockTrabRepo.buscarPorId("t1")).thenReturn(trabalho);
        when(mockAlunoRepo.buscarPorId("a2")).thenReturn(avaliador1);
        when(mockAlunoRepo.buscarPorId("a3")).thenReturn(avaliador2);
        
        // 4. Injetar mocks no Application (INVERSÃO DE DEPENDÊNCIA!)
        IGerenciadorAvaliacoes gerenciador = new GerenciadorAvaliacoes(mockTrabRepo, mockAlunoRepo);
        
        // === ACT (Agir) ===
        
        // 5. Executar casos de uso SEM banco real!
        gerenciador.avaliarTrabalho("n1", "t1", "a2", 9.0);
        gerenciador.avaliarTrabalho("n2", "t1", "a3", 7.0);
        double media = gerenciador.consultarMedia("t1");
        
        // === ASSERT (Verificar) ===
        
        // 6. Verificar resultado
        assertEquals(8.0, media, 0.01, "Média de 9.0 e 7.0 deve ser 8.0");
        
        // 7. Verificar que repositório foi chamado (optional)
        verify(mockTrabRepo, times(3)).buscarPorId("t1");  // 2x avaliar + 1x consultar
        verify(mockTrabRepo, times(2)).salvar(trabalho);   // 2x ao avaliar
        
        // ✅ Teste passou! Sem banco de dados, sem infraestrutura!
    }
    
    @Test
    public void deveLancarExcecaoSeTrabalhoNaoExistir() {
        // === ARRANGE ===
        ITrabalhoRepositorio mockTrabRepo = mock(ITrabalhoRepositorio.class);
        IAlunoRepositorio mockAlunoRepo = mock(IAlunoRepositorio.class);
        
        // Simular que trabalho NÃO existe
        when(mockTrabRepo.buscarPorId("t999")).thenReturn(null);
        
        IGerenciadorAvaliacoes gerenciador = new GerenciadorAvaliacoes(mockTrabRepo, mockAlunoRepo);
        
        // === ACT & ASSERT ===
        Exception exception = assertThrows(IllegalArgumentException.class, () -> {
            gerenciador.consultarMedia("t999");
        });
        
        assertTrue(exception.getMessage().contains("não encontrado"));
        
        // ✅ Validação funcionou corretamente!
    }
}

// ✅ VANTAGENS DESSES TESTES:
// - Rápidos (sem I/O, sem banco)
// - Isolados (não dependem de estado externo)
// - Determinísticos (sempre mesmo resultado)
// - Fáceis de escrever (mocks triviais)

// ❌ NA ARQUITETURA EM CAMADAS:
// Esses testes seriam IMPOSSÍVEIS ou muito mais difíceis!
// Application instanciava repositórios internamente → não dava para passar mocks.

✅ Por que isso só é possível com Hexagonal?

Na arquitetura em camadas, GerenciadorAvaliacoes instanciava os repositórios dentro dele: 

// ❌ Camadas - NÃO dá para testar facilmente
public class GerenciadorAvaliacoes {
    private TrabalhoRepositorio repo = new TrabalhoRepositorio();  // ❌ Hardcoded!
    // Não dá para passar mock!
}

Com Hexagonal, recebemos interfaces via construtor (Dependency Injection): 

// ✅ Hexagonal - FÁCIL testar
public class GerenciadorAvaliacoes {
    private ITrabalhoRepositorio repo;
    
    public GerenciadorAvaliacoes(ITrabalhoRepositorio repo) {  // ✅ Injeção!
        this.repo = repo;
    }
}
// Agora podemos passar mocks nos testes!

📝 Resumo: Camadas → Hexagonal

🔄 A Grande Mudança

A transição de Arquitetura em Camadas para Hexagonal é essencialmente sobre INVERTER DEPENDÊNCIAS usando INTERFACES.

📐 Estrutura:

Camada O que mudou
Domain (Model) ✅ Mantido igual (Aluno, Trabalho, Nota)
Domain (Ports) 🆕 NOVIDADE! Interfaces (IAlunoRepositorio, ITrabalhoRepositorio, IGerenciadorAvaliacoes)
Application 🔄 Agora IMPLEMENTA interface e DEPENDE de interfaces (não de classes concretas)
Adapters (Output) 🔄 Repositórios agora IMPLEMENTAM interfaces do Domain
Adapters (Input) 🔄 Controller agora INJETA dependências e chama interfaces

🎯 Checklist de Conversão:

  • ✅ Criar interfaces (Ports) no Domain
  • ✅ Application implementa Driving Port (IGerenciadorAvaliacoes)
  • ✅ Application depende de Driven Ports (IAlunoRepositorio, ITrabalhoRepositorio)
  • ✅ Repositórios implementam Driven Ports
  • ✅ Controller injeta Adapters no Application
  • ✅ Controller chama Application através da Driving Port

🏆 Resultado Final:

  • ✅ Domain 100% independente de tecnologias
  • ✅ Application facilmente testável (mocks)
  • ✅ Adapters intercambiáveis (trocar HashMap por SQL)
  • ✅ SOLID principles respeitados (especialmente DIP)
  • ✅ Sistema flexível e manutenível
